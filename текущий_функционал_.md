# Справка по текущему функционалу проекта

Документ описывает фактически реализованный функционал backend и frontend на текущий момент. Проект представляет собой систему электронной очереди с клиентским интерфейсом, рабочим местом оператора и административной панелью.

## Общая структура

В проекте есть три пользовательских интерфейса:

- клиентский интерфейс: `/c/:queueId`;
- интерфейс оператора: `/o`;
- интерфейс администратора: `/a`.

Технологический стек:

- backend: Django, Django REST Framework, drf-spectacular, PostgreSQL, prometheus-client, pywebpush;
- frontend: React, Vite, TypeScript, Ant Design, React Query, Zustand, i18next.

Основной API находится под префиксом `/api/v1/`.

## Backend

Backend реализован как Django-проект с несколькими доменными приложениями:

- `users` - пользователи, роли, авторизация операторов и администраторов;
- `clients` - клиенты очереди;
- `companies` - компании и филиалы;
- `queues` - очереди и талоны;
- `notifications` - обращения, жалобы, клиентские подписки на уведомления и webhook-интеграции;
- `analytics` - технические HTTP-метрики и бизнес-метрики очередей;
- `core` - настройки проекта, обработчик ошибок, middleware метрик.

К `/api/v1/` сейчас подключены приложения:

- `users`;
- `clients`;
- `companies`;
- `queues`;
- `notifications`;
- `analytics`.

Также доступны:

- `/api/v1/schema/` - OpenAPI schema;
- `/api/v1/docs/` - Swagger UI.

## Модели данных backend

Реализованы следующие сущности:

- `User` - пользователь системы;
- `OperatorToken` - токен оператора;
- `AdminToken` - токен администратора;
- `Company` - компания;
- `Branch` - филиал компании;
- `Queue` - очередь;
- `Ticket` - талон в очереди;
- `Client` - клиент очереди;
- `FeedbackItem` - обращение или жалоба;
- `ClientNotificationSubscription` - подписка клиента на web-push или VK-уведомления;
- `WebhookSubscription` - подписка внешней системы на события очереди;
- `WebhookDelivery` - журнал доставки webhook-событий;

У пользователя есть роли:

- `AD` - администратор;
- `OPR` - оператор.

У талона есть статусы:

- `WAITING` - ожидает;
- `CALLED` - вызван;
- `IN_SERVICE` - обслуживается;
- `COMPLETED` - завершен;
- `SKIPPED` - пропущен;
- `LEFT` - клиент ушел;
- `NOT_ARRIVED` - клиент не подошел вовремя;
- `REMOVED` - талон удален оператором.

## Авторизация

Реализована раздельная авторизация для оператора и администратора.

Endpoint'ы оператора:

- `POST /api/v1/auth/operator/login/` - вход оператора;
- `GET /api/v1/auth/operator/me/` - получение текущей сессии оператора;
- `POST /api/v1/auth/operator/logout/` - выход оператора;
- `PATCH /api/v1/auth/operator/settings/` - обновление настроек оператора.

Endpoint'ы администратора:

- `POST /api/v1/auth/admin/login/` - вход администратора;
- `GET /api/v1/auth/admin/me/` - получение текущей сессии администратора;
- `POST /api/v1/auth/admin/logout/` - выход администратора;
- `PATCH /api/v1/auth/admin/settings/` - обновление настроек администратора.

Авторизация использует кастомные bearer-токены. Токены живут 12 часов. При повторном входе старые токены пользователя удаляются. Есть мягкая миграция старых пользователей с plaintext-паролем на Django password hash.

## Очереди и талоны

Backend реализует основную бизнес-логику электронной очереди:

- создание очереди;
- редактирование очереди;
- получение snapshot очереди;
- постановка клиента в очередь;
- генерация номера талона в формате `Q{queueId}-0001`;
- запрет второго активного талона у одного клиента в одной очереди;
- лимит количества клиентов в очереди;
- вызов следующего талона;
- приглашение конкретного талона;
- перевод талона в обслуживание;
- завершение обслуживания;
- возврат талона в очередь;
- отметка неявки клиента;
- пропуск одного человека вперед;
- выход клиента из очереди;
- удаление талона оператором;
- массовое удаление талонов из очереди;
- автоматическое истечение таймера вызванного талона;
- сохранение ответственного оператора талона для последующей аналитики;
- отправка клиентских уведомлений при вызове талона;
- отправка webhook-события `ticket.status_changed` при любом изменении статуса талона.

Если талон находится в статусе `CALLED` дольше заданного таймера, он переводится в статус `NOT_ARRIVED`.

Основные endpoint'ы:

- `GET /api/v1/queues/:id/` - публичное получение очереди;
- `GET /api/v1/queues/:id/snapshot/` - snapshot очереди;
- `POST /api/v1/queues/:id/invite-next/` - вызвать следующий талон;
- `POST /api/v1/queues/:id/tickets/delete/` - массовое удаление талонов;
- `GET /api/v1/admin/queues/` - список очередей компании администратора;
- `POST /api/v1/admin/queues/` - создание очереди;
- `PATCH /api/v1/admin/queues/:id/` - обновление очереди;
- `DELETE /api/v1/admin/queues/:id/` - удаление очереди;
- `GET /api/v1/operator/queues/` - список очередей, назначенных оператору;
- `PATCH /api/v1/operator/queues/:id/settings/` - изменение разрешенных настроек назначенной очереди;
- `GET /api/v1/tickets/` - список талонов;
- `GET /api/v1/tickets/:id/` - получение талона;
- `POST /api/v1/tickets/join/` - встать в очередь;
- `PATCH /api/v1/tickets/:id/status/` - изменить статус талона;
- `POST /api/v1/tickets/:id/append-to-queue/` - вернуть талон в очередь;
- `POST /api/v1/tickets/:id/skip-one-ahead/` - пропустить одного вперед;
- `POST /api/v1/tickets/:id/invite/` - пригласить конкретный талон;
- `POST /api/v1/tickets/:id/remove/` - удалить талон из очереди.

Доступ к управлению талонами защищен bearer-токеном оператора. Оператор может выполнять действия только с талонами и очередями, которые явно назначены ему через связь `assigned_queues`. Для клиента без токена доступно только самостоятельное вступление в очередь, получение snapshot, выход из очереди и пропуск одного человека вперед.

Snapshot очереди возвращает:

- id и название очереди;
- язык очереди;
- количество ожидающих;
- текущий талон;
- список ожидающих талонов;
- талон текущего клиента, если передан `client_id`;
- признаки, что клиент обслужен, удален или не подошел;
- оставшееся время для вызванного талона;
- настройки таймера вызванного талона;
- примерное время ожидания клиента в секундах, если есть история завершенных обслуживаний.

## Обратная связь

В backend есть модель обратной связи `FeedbackItem`.

Поддерживаются типы:

- `FEEDBACK`;
- `COMPLAINT`.

Поддерживаются статусы:

- `NEW`;
- `IN_PROGRESS`;
- `RESOLVED`.

Есть backend ViewSet для просмотра и удаления обращений администратором:

- `/api/v1/admin/feedback/`.

Для клиентского интерфейса есть публичный endpoint:

- `POST /api/v1/feedback/` - создать обращение клиента по `queue_id`.

Доступ к обращениям защищен токеном администратора. Администратор видит только обращения своей компании. Создание и редактирование обращений в кабинете администратора отключены: администратор просматривает обращения клиентов и может удалить ошибочную или неактуальную запись. API `/api/v1/admin/feedback/` поддерживает `GET`, `GET :id` и `DELETE`, но не поддерживает `POST`, `PUT` и `PATCH`.

Публичное клиентское создание обращения не принимает компанию и филиал из браузера. Backend находит очередь по `queue_id` и сам привязывает обращение к компании, филиалу и очереди.

Обращение может содержать оценку обслуживания от 1 до 5 звезд и ссылку на талон. Если отзыв связан с завершенным талоном, оценка учитывается в аналитике оператора.

## Уведомления и webhook-интеграции

В backend реализована подсистема клиентских уведомлений для вызова талона и webhook-интеграций для изменений статуса талона.

Каналы клиентских уведомлений:

- web-push через браузерную Push API подписку;
- локальные браузерные уведомления по polling-состоянию страницы, если VAPID-ключи web-push не настроены;
- VK-уведомления по VK ID клиента, полученному через OAuth-подключение VK.

Подписки клиента хранятся в `ClientNotificationSubscription`. Для web-push сохраняются endpoint, `p256dh` и `auth` ключи браузерной подписки. Для VK сохраняется `vk_user_id`, а у клиента дополнительно обновляются `vk_id` и флаг `send_notification`. Ручной endpoint подключения VK ID сохранен на backend, но клиентский интерфейс использует кнопку OAuth-подключения VK.

При любом изменении статуса талона backend в одном месте планирует обработку события после успешного commit транзакции. Для webhook-интеграций формируется основное событие `ticket.status_changed` с полями `previous_status` и `new_status`. Дополнительно поддерживается совместимое событие `ticket.called`: оно отправляется только при переходе талона в статус `CALLED` для подписок, которые явно выбрали `ticket.called` и не выбрали `ticket.status_changed`.

При переводе талона в статус `CALLED` дополнительно выполняются клиентские уведомления:

- выполняется попытка отправки web-push через `pywebpush`, если заданы `WEB_PUSH_VAPID_PRIVATE_KEY` и `WEB_PUSH_VAPID_SUBJECT`;
- выполняется попытка отправки сообщения VK через `messages.send`, если задан `VK_BOT_ACCESS_TOKEN`;
- при отсутствии ключей или токена доставка фиксируется в логах и Prometheus как `SKIPPED`, не ломая локальную разработку.

Если публичный VAPID-ключ отсутствует, frontend не показывает ошибку "Push-ключи не настроены на сервере", а включает локальные браузерные уведомления через `Notification API`. Такой режим работает, пока клиентская страница очереди открыта и получает обновления через polling. Для настоящего фонового web-push после закрытия вкладки по-прежнему требуются VAPID-ключи на backend.

На мобильных устройствах frontend проверяет поддержку web-push до запроса подписки. Для включения web-push страница должна быть открыта в secure context, то есть по HTTPS или на localhost. Для iPhone и iPad дополнительно требуется открыть QueueFlow как PWA-приложение, добавленное на экран "Домой"; для этого во frontend добавлен web app manifest. Запрос системного разрешения на уведомления выполняется сразу после нажатия кнопки, чтобы мобильный браузер не потерял пользовательский жест.

Для внешних интеграций есть webhook-подписки `WebhookSubscription`. Администратор компании может создать подписку на все очереди компании или на конкретную очередь. Подписка содержит URL, список событий (`ticket.status_changed`, `ticket.called`), активность и необязательный секрет. При наличии секрета запрос подписывается заголовком `X-QueueFlow-Signature` через HMAC-SHA256. Попытки webhook-доставок сохраняются в `WebhookDelivery`.

Endpoint'ы уведомлений:

- `GET /api/v1/notifications/web-push/public-key/` - получить публичный VAPID-ключ;
- `POST /api/v1/notifications/web-push/subscribe/` - подписать клиента на web-push;
- `POST /api/v1/notifications/vk/oauth/start/` - начать OAuth-подключение VK для клиента очереди;
- `GET /api/v1/notifications/vk/oauth/callback/` - callback VK OAuth, сохраняющий VK ID клиента;
- `POST /api/v1/notifications/vk/subscribe/` - подключить VK ID клиента;
- `GET /api/v1/notifications/status/` - получить статус активных подписок клиента;
- `GET /api/v1/admin/webhook-subscriptions/` - список webhook-подписок компании;
- `POST /api/v1/admin/webhook-subscriptions/` - создать webhook-подписку;
- `PATCH /api/v1/admin/webhook-subscriptions/:id/` - обновить webhook-подписку;
- `DELETE /api/v1/admin/webhook-subscriptions/:id/` - удалить webhook-подписку.

Публичные endpoint'ы клиентских подписок не требуют bearer-токена, так как вызываются с клиентской страницы очереди. Управление webhook-подписками защищено токеном администратора и ограничено компанией администратора.

Для VK OAuth используются настройки `VK_OAUTH_CLIENT_ID`, `VK_OAUTH_CLIENT_SECRET`, `VK_OAUTH_REDIRECT_URI`, `VK_OAUTH_STATE_MAX_AGE_SECONDS` и `VK_BOT_URL`. После успешного OAuth callback сохраняет VK ID клиента, а при заданном `VK_BOT_URL` открывает диалог с ботом, чтобы клиент мог разрешить сообщения от сообщества. Отправка сообщений выполняется токеном `VK_BOT_ACCESS_TOKEN`.

## Метрики

В backend есть middleware `PrometheusMetricsMiddleware`, который считает:

- количество HTTP-запросов;
- метод;
- endpoint;
- HTTP status code;
- company ID;
- latency запроса.

Метрики также сохраняются в журнал `ApiRequestLog`, чтобы можно было считать технические показатели по времени, филиалу, очереди и оператору.

Метрики агрегируются для администратора по его компании.

Endpoint:

- `GET /api/v1/admin/metrics/`.

Поддерживаемые query-фильтры:

- `branch_id` - фильтр по филиалу;
- `queue_id` - фильтр по очереди;
- `operator_id` - фильтр по оператору;
- `date_from` - начало периода;
- `date_to` - конец периода.

Возвращаются:

- технические метрики: всего запросов, количество ошибочных запросов, процент ошибок API, средняя задержка, статистика по endpoint'ам;
- бизнес-метрики очередей: активные, ожидающие, вызванные, обслуживаемые, завершенные, не подошедшие, ушедшие и удаленные талоны;
- среднее время ожидания и среднее время обслуживания;
- максимальное ожидание;
- процент ушедших клиентов, неявок, удалений и завершенных обслуживаний;
- загруженность по лимитам очередей;
- нагрузка на активного оператора;
- производительность обслуживаний в час;
- SLA ожидания до 10 минут;
- средняя оценка обслуживания;
- количество отзывов, оценок и жалоб;
- отчет по филиалам;
- отчет по очередям;
- динамика по дням;
- пиковые часы постановки в очередь;
- загрузка операторов по назначенным очередям.

## Frontend

Frontend разделен на три приложения:

- `client`;
- `operator`;
- `admin`.

Глобальная маршрутизация:

- `/c/*` - клиентское приложение;
- `/o/*` - операторское приложение;
- `/a/*` - административное приложение;
- `/_debug` - debug page.

API base URL настраивается:

- через переменную окружения `VITE_API_URL`;
- fallback для локальной разработки: `http://localhost:8000/api/v1`.

Авторизационный токен автоматически подставляется Axios interceptor'ом:

- для `/auth/operator/` и `/operator/` используется токен оператора;
- для `/auth/admin/` и `/admin/` используется токен администратора;
- для остальных запросов берется токен оператора или администратора, если он есть.

## Клиентский интерфейс

Маршрут:

- `/c/:queueId`.

Реализовано:

- загрузка snapshot очереди;
- polling snapshot очереди раз в 1 секунду;
- создание и хранение `device_id`;
- создание и хранение `queue_token`;
- восстановление сессии из `sessionStorage`;
- экран гостя;
- отображение количества людей в очереди;
- постановка в очередь;
- экран активного талона;
- отображение номера талона;
- отображение количества людей перед клиентом;
- отображение текущего вызванного или обслуживаемого талона;
- отображение общего количества ожидающих;
- отдельное состояние активного талона при обслуживании (`IN_SERVICE`);
- отображение прогресса ожидания;
- отображение примерного времени ожидания, если backend смог рассчитать прогноз;
- отображение таймера, когда талон вызван;
- кнопка "Включить уведомления" над действием пропуска;
- модальное окно настройки уведомлений;
- включение web-push через service worker при настроенных VAPID-ключах;
- fallback на локальные браузерные уведомления без VAPID-ключей;
- проверка HTTPS/secure context для web-push;

Если backend сообщает, что клиент обслужен, frontend переводит пользователя на страницу выхода. Если backend сообщает, что клиент не подошел вовремя, frontend переводит пользователя на страницу missed.

## Инфраструктура

Для контейнерного запуска есть:

- `Dockerfile`;
- корневой `docker-compose.yml`, который поднимает единый комплект backend, frontend, PostgreSQL и Prometheus;
- production compose-файл `deploy/compose.prod.yml`, рассчитанный на запуск опубликованных образов из Docker registry;
- shell-скрипты запуска backend и deploy-скрипты для сервера;
- PostgreSQL-сервис `db` в Docker Compose с изолированным именованным volume;
- образ PostgreSQL зафиксирован на major-версии 16;
- backend подключается к БД внутри Docker Compose по `db:5432`.

Backend-образ запускает Django через Gunicorn, применяет миграции при старте контейнера и собирает static-файлы для production-режима. Production-настройки Django читаются из переменных окружения: `DJANGO_ALLOWED_HOSTS`, `DJANGO_CSRF_TRUSTED_ORIGINS`, `DJANGO_CORS_ALLOWED_ORIGINS`, secure-cookie флаги, PostgreSQL-настройки и интеграционные ключи уведомлений.

В frontend есть:

- `npm run dev`;
- `npm run build`;
- `npm run lint`;
- `npm run preview`.
- `Dockerfile`, который собирает Vite-приложение и отдает результат через Nginx;
- `frontend/nginx.conf`, который обслуживает SPA и проксирует `/api/v1/`, `/admin/` и `/static/` в backend.

Для CI/CD добавлен GitHub Actions workflow `.github/workflows/docker-publish.yml`. При push в `main` он собирает и публикует Docker-образы backend и frontend в GitHub Container Registry:

- `ghcr.io/alexander03e/qr_app-backend:main`;
- `ghcr.io/alexander03e/qr_app-frontend:main`.

В production compose добавлен Watchtower для автоматического обновления контейнеров из Docker registry. Автообновление включено только для сервисов `backend` и `frontend` через label `com.centurylinklabs.watchtower.enable=true`, поэтому PostgreSQL не обновляется автоматически.

Для первичного и последующих развертываний добавлена инструкция `DEPLOY.md`. Серверный Nginx-конфиг для домена `cfifeg1.fvds.ru` хранится в `deploy/nginx/cfifeg1.fvds.ru.conf`.

## Уже есть, но выглядит неполным или требует проверки

1. SMS-уведомления не реализованы; рабочие внешние каналы сейчас web-push, локальные браузерные уведомления, VK и webhook.
2. Публичные клиентские endpoint'ы работают без bearer-токена; нужно отдельно зафиксировать в дипломе, какие из них являются намеренно публичными.
3. В проекте есть backend-тесты, но нет отдельного набора frontend/e2e-тестов для операторского и административного сценариев.
4. Бизнес-метрики считаются по текущим данным в БД; для дипломной демонстрации может понадобиться seed-скрипт с реалистичной историей талонов.
5. Локальные браузерные уведомления без VAPID работают только при открытой клиентской странице; фоновые push-уведомления требуют настроенных `WEB_PUSH_VAPID_PUBLIC_KEY` и `WEB_PUSH_VAPID_PRIVATE_KEY`.
